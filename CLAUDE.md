# Claude作業指針

## 最重要原則：自律的に動く

- **ユーザーに何かを「やってもらう」のは最後の最後の手段**
- 環境調査・ファイル確認・代替手段の模索を先にやりきる
- 「〜を教えてください」「〜はできますか？」と聞く前に、自分で調べる
- 詰まったら別のアプローチを試す。すぐユーザーに投げない

## 分析・報告

- データが少ない・APIが使えない場合でも、手元にあるデータで最大限の分析をする
- 「できない理由」を並べるより「今できること」をやる
- 口頭・会話で得た情報も積極的に活用する

## 作業スタイル

- 問題が起きたら原因を特定してから対処する
- ブロッカーがあれば迂回路を探す（crontabがなければ/etc/cron.dを使う、など）
- ユーザーが言及した過去の指示・方針はCLAUDE.mdに随時追記する

## 安全運用版の前提

- このリポジトリは、参照元の記事の構成をベースにしつつ、**安全運用を優先した派生版**として扱う
- GitHub Actions を主経路にする。出先からの確認・承認・フィードバックは Actions + Discord で完結させる
- 記事の「高頻度連投」や「深夜帯の大量投稿」は再現しない
- 投稿はスロット制を前提にする。朝・昼・夜の少数枠に絞り、bot っぽい挙動を避ける
- 分析 cron と生成 cron は分離する。分析は `fetcher -> analyst -> supervisor` で回し、Discord に転送した結果を見て同日中に `operator_feedback.json` に戻せる時間を確保する
- 分析は 11:00 JST 前後に回し、`fetcher` の 24h 窓に引っかからないようにする。生成は 14:00 JST 前後に回してレビュー反映の余白を残す
- 生成 cron はレビュー時間のあとに `researcher -> writer` を回し、直近の analysis と operator feedback を次回生成に反映する
- レビューは「止めるためのゲート」として使う。こちらは却下理由を次回生成に反映するループまで接続済み
- 生成・投稿・分析は分離して扱い、どこか1つが壊れても全体停止しないように確認する
- 記事に書かれた仕組みと実装が一致しない場合は、まず「安全運用のための意図的な差分」かどうかを確認する
